     Plan: Bundle System + Player Spawn on Connect

     Context

     We have a working server/client WebSocket connection in _game_rts. The
     next step is spawning a player entity (cube) when a client connects.
     This requires introducing the "bundle" concept ‚ÄĒ a function that spawns
      an entity with its components, registered by numeric ID so client and
     server agree on what to spawn but can have different component sets
     (server: game state only, client: adds rendering).

     Key Design Decisions

     - Bundle: () => ConductEntity ‚ÄĒ spawns entity, adds components, returns
      entity ID
     - Registry: Simple Map<number, Bundle> in @conduct/networking, apps
     register their bundles at startup
     - Networked component: Replaces ReplicatedTag, adds bundleId field so
     snapshots carry the bundle identity
     - Serialization keys: Use component class .name (string) instead of
     numeric ComponentId ‚ÄĒ the runtime IDs differ between client/server
     processes
     - Deferred spawns: ConductAddComponent is deferred, so snapshot data
     for newly-spawned entities is applied one tick later via
     flushPendingNetworkUpdates()

     File Changes

     1. Modify packages/networking/src/replication.ts

     - Replace ReplicatedTag (empty class) with Networked { bundleId = 0 }
     - Change replication registry from Map<number, ReplicableComponent>
     (keyed by ComponentId) to Map<string, ReplicableComponent> (keyed by
     class .name)
     - Add getReplicatedComponent(name) lookup
     - Add Bundle type, ConductRegisterBundle(id, bundle), getBundle(id)

     2. Create packages/networking/src/snapshot.ts

     - generateSnapshot(entities) ‚ÄĒ server-side, iterates tracked entities,
     reads replicated components via ConductGetComponent, returns Snapshot
     - applySnapshot(snapshot) ‚ÄĒ client-side, for new entities: reads
     bundleId from Networked data, calls client bundle, creates entity
     mapping. For existing entities: applies component data via
     ConductSetComponent
     - flushPendingNetworkUpdates() ‚ÄĒ applies deferred component data for
     entities spawned last tick (called from a FixedUpdate system)

     3. Modify packages/_game_rts/src/shared/index.ts

     - Export BUNDLE = { PLAYER: 1 } as const
     - Call ConductNetworkReplicateComponent(Transform3D) to register shared
      replicated components

     4. Modify packages/_game_rts/src/server/index.ts

     - Import shared (triggers component registration)
     - Register server player bundle: Transform3D + Networked
     - On connect: spawn player entity via bundle, track in
     networkedEntities array
     - Register a FixedUpdate system that generates + broadcasts snapshots
     each tick

     5. Modify packages/_game_rts/src/client/index.ts

     - Import shared (triggers component registration)
     - Register client player bundle: Transform3D + Networked (rendering
     components can be added later when renderer is wired up)
     - Handle snapshot messages by calling applySnapshot()
     - Register a FixedUpdate system that calls flushPendingNetworkUpdates()

     No changes needed

     - packages/networking/src/protocol.ts ‚ÄĒ SerializedEntity.components
     already uses Record<string, ...>, compatible
     - packages/networking/src/entityMap.ts ‚ÄĒ used as-is by snapshot.ts

     Verification

     1. Build networking + server packages: pnpm -r --filter 
     @conduct/networking --filter @conduct/server --filter 
     @conduct/simulation build
     2. Start server: pnpm --filter @conduct/game-rts dev:server ‚ÄĒ should
     log "listening on ws://localhost:3001"
     3. Start client: pnpm --filter @conduct/game-rts dev:client ‚ÄĒ open
     browser
     4. Check server logs: "player 1 connected" + snapshot broadcast each
     tick
     5. Check browser console: "assigned player ID: 1" + entity spawned via
     bundle (logged)
     6. Open second tab: both clients should receive snapshots with all
     player entities
